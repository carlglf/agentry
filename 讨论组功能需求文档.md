# 讨论组（Discussion Group）功能需求文档

> 本文档记录在 Agent Console 现有架构之上新增「讨论组」功能的需求与设计决策。
> 配套阅读：根目录《Agent Console 控制台需求文档.pdf》、`CLAUDE.md`（架构总览）。
> 状态：核心讨论组能力已实现；「用户反馈驱动的派生讨论」待实现（11 项关键决策见第 8 节）。后端逻辑在 `agent-console/server/*.mjs`、CLI 在 `agent-console/bin/acg.mjs`、
> 前端在 `agent-console/src/App.tsx`（讨论组树/会话流/成员终端标签）；测试在 `agent-console/test/*.test.mjs`（`npm test`）。
> 实现期补充决策：maxRounds 按发言次数计；暂不做 `AGENT_CONSOLE_TOKEN`；`acg recap` 由 server 直接拼接；单文件 JSON 原子写。

## 1. 背景与目标

现有 Agent Console 中，每个 Agent 是一个独立的 PTY 会话、独立上下文，彼此**看不到对方的输出**。
本功能在项目下新增一类节点「讨论组」：定义一套**可复用的带人设成员阵容**，由一名**主理人**启动并判断结束。
讨论组与话题**解耦**——同一组可先后/并发讨论多个话题，每个话题是一次独立的「话题讨论（DiscussionSession）」，
成员围绕该话题多轮自驱讨论。为支撑 agent 之间的接力发言与可控收尾，新增一个供 agent 在 PTY 内调用的 **CLI 工具**。

核心价值：把"多 agent 协作讨论"变成一个可创建、可观察、可终止的一等对象，而不是人工在多个终端间来回复制粘贴。

## 2. 核心概念与数据模型

> **核心解耦**：讨论组与话题分离。讨论组是**可复用的人马阵容/模板**（成员 + 人设 + 主理人 + 规则），
> 不绑定具体话题；每次围绕一个话题的具体讨论，是一个独立的 **DiscussionSession（话题讨论）**。
> 同一个讨论组可以先后/并发讨论多个话题。

### 2.1 讨论组 DiscussionGroup（可复用模板）
- 挂在**项目（Project）下**，与现有 Agent 在左侧树中**平级**。
- 定义 **N 个讨论成员（member）+ 人设 + 职责 + 规则**，**不含 topic / 运行态**。
- 成员中**有且只有一个主理人（host）**——创建时校验：host 数量必须等于 1。
- 可被复用于多个话题讨论。

> **存储位置（决策 9）**：与现有"全部状态浏览器本地"不同，讨论组/话题讨论/消息**持久化在 server 端 JSON 文件**，
> 前端通过 API 读取。原因：CLI 被打包成全局 npm bin（`acg serve` 起服务），讨论记录跨重启保留、可多端查看更自然。
> 这是对现有架构的一处有意偏离——现有 projects/agents/commands 仍走前端 zustand persist，互不影响。

字段定义：

```ts
type DiscussionGroup = {
  id: string;
  projectId: string;
  name: string;
  rule: string;             // 注入给每个成员的讨论规则/约束（含发言长度、礼仪等）
  createdAt: string;
  updatedAt: string;
};

type DiscussionMember = {
  id: string;
  groupId: string;
  name: string;             // 成员显示名（也是 handoff 提名时的标识，需组内唯一）
  runtime: AgentRuntime;    // codex | claude
  model: string;
  persona: string;          // 人设
  duty: string;             // 职责
  isHost: boolean;          // 组内恰好一个为 true
};
```

### 2.2 话题讨论 DiscussionSession（一次具体讨论）
一个 DiscussionSession 绑定**恰好一个话题**，承载该话题讨论的全部运行态与消息。

```ts
type DiscussionSession = {
  id: string;
  groupId: string;          // 复用哪个讨论组的阵容
  topic: string;            // 本次讨论的话题（单一）
  parentSessionId?: string; // 从哪一轮结论派生；首轮为空
  revisionNo: number;       // 线性讨论轮次，首轮为 1
  reopenMode?: "revise" | "restart"; // 修订旧结论，或从头讨论
  userFeedbackMessageId?: string;      // 触发本轮的用户意见
  previousConclusionSnapshot?: string; // 上轮结论快照
  status: "idle" | "running" | "ended";
  maxRounds: number;        // 硬上限，防失控/token 爆炸（按话题设定）
  currentMemberId: string;  // 当前持有发言权的成员
  round: number;            // 已进行轮次
  createdAt: string;
  updatedAt: string;
};
```

### 2.3 讨论消息 DiscussionMessage
讨论过程的权威记录由 **server 端维护**（agent 之间不共享上下文，必须由 server 中继）。**消息归属到 session**。

```ts
type DiscussionMessage = {
  id: string;
  sessionId: string;        // 归属的话题讨论
  type: "member_message" | "user_feedback";
  memberId?: string;        // Agent 发言者；user_feedback 时为空
  seq: number;              // 【按 session】单调递增序号，用于计算"增量"
  content: string;          // Agent 发言或用户提交的原始反馈
  createdAt: string;
};
```

### 2.4 成员会话模型决策 ✅【决策 3+5：每个话题各自新开专属会话】
- 每发起一个话题讨论（DiscussionSession），**为每个成员拉起该 session 专属的 PTY 会话**（带各自 persona/duty）。
- **话题之间记忆完全隔离**、互不污染；同一讨论组可**并发**讨论多个话题（各自独立会话）。
- 与项目里独立的 Agent 也互不干扰。
> 实现含义：成员不是对现有 `Agent` 的引用，而是讨论组的成员配置；运行期会话按 **session 维度**实例化。
> 复用 `startPtySession` 的能力，但 session key 需带 sessionId + memberId 维度
> （如 `disc:<sessionId>:member:<memberId>`），避免与现有 `sessions` map 的 agentId 撞键。

**会话回收 ✅【决策 8：保留 + 主动关闭】**：
- 话题讨论 `end` 后，成员 PTY 会话**默认保留存活**（不自动 kill、无空闲 TTL）。
- 已正常结束的 session **不再原地重开**；用户对结果有意见时，按第 9 节创建派生 session，父 session 始终保持 `ended`。
- `reopen` 仅作为异常中断后恢复同一 session 的内部能力，不作为正常结果页操作。
- 提供**显式"关闭会话"操作**（UI 按钮 + 可选 CLI），主动 kill 该话题的全部成员 PTY、释放资源。
- 注意：PTY 进程无法跨 `acg serve` 重启存活；异常恢复时需重建 PTY 并回灌讨论上下文。

## 3. 调度与生命周期

> 生命周期以 **DiscussionSession（话题讨论）** 为单位：每个话题讨论独立启动、调度、结束。

### 3.1 发言调度决策 ✅【决策 2：agent 提名 + 主理人随时结束 + 硬上限】
- **启动**：用户为某讨论组发起一个话题 → 创建 DiscussionSession → 为成员拉起专属会话 → 主理人首发。
  server 向主理人会话注入开场 prompt（话题 + 人设 + 职责 + 规则 + CLI 用法说明）。
- **接力**：每位成员发言后，用 CLI **提名下一位**发言人（`--next <成员名>`）。
- **结束**：主理人在轮到自己时可调用 `acg end` 结束本话题讨论；UI 也提供"强制结束"按钮（人工兜底）。
- **硬上限**：达到该 session 的 `maxRounds` 自动结束，防止 agent 互相提名无限循环 / token 爆炸。

**调度状态机（server 端，每个 session 一份）**：
```
idle --(用户发起话题/点击"开始")--> running：拉起成员会话，注入开场 prompt 给 host
running 中，收到某成员的 say 回调：
   1. 记录 DiscussionMessage（该 session 内 seq++）
   2. 若带 --next X：把"自 X 上次发言以来的增量消息"注入 X 的会话（见 4.2 / 5）
   3. 若带 end（仅 host 可）：status 置 ended，停止注入
   4. round 达到 maxRounds：status 置 ended
ended：不再注入；成员会话默认保留存活，直到用户显式"关闭会话"才 kill（决策 8）。
用户对结果有意见时创建新的派生 session，不修改该 ended session。
```

### 3.2 校验与边界规则
- **提名校验**：`--next` 必须是组内存在的成员名；不可提名已结束的组；提名自己需要明确允许或拒绝（默认拒绝，避免自说自话刷轮次）。
- **越权校验**：`acg end` 仅主理人有效；非主理人调用应被拒绝并提示。
- **丢轮防护**：server 必须保证每个增量都送达目标会话，否则该成员记忆出现"空洞"。投递失败需重试/告警。
- **并发**：理论上同一时刻只有"当前发言人"在说话；server 以"当前持有发言权的 memberId"为准，拒绝非当前发言人的 say（或排队）。

## 4. CLI 工具规格

### 4.1 形态与身份 ✅【决策 6：全局 npm bin + 自带 serve】
- 整个 agent-console **打包成可全局安装的 npm 包**，暴露一个 bin：`acg`。
- `acg` 既是**应用入口**也是**讨论 CLI**：
  - `acg serve` —— 启动 web server（替代现在的 `npm start` / `node server.mjs`），用户用它起服务。
  - `acg say / end / ...` —— agent 在 PTY 内调用，驱动讨论（见 4.2）。
- CLI 是 agent 在自己 PTY 内调用的本地命令（agent 处于 YOLO/bypass 模式，本就能跑 shell）。
- 命令名定为 `acg`。
- **身份通过启动 PTY 时注入的环境变量传递**（沿用现有 `AGENT_CONSOLE` / `AGENT_CONSOLE_YOLO` 注入点，见 `server.mjs:201-206`）：
  - `AGENT_CONSOLE_SESSION_ID`（话题讨论 id，发言/结束都挂在它下面）
  - `AGENT_CONSOLE_GROUP_ID`
  - `AGENT_CONSOLE_MEMBER_ID`
  - `AGENT_CONSOLE_API`（本地 server 地址，如 `http://127.0.0.1:5173`）
  - 可选 `AGENT_CONSOLE_TOKEN`（防误调用/越权）
- CLI 通过本地 HTTP 回调 server，沿用现有 `/api/` REST 风格。

### 4.2 命令集（决策 1：CLI 显式携带发言内容）✅
> **关键决策**：agent 把发言内容作为参数/stdin **显式提交**，server **完全不再依赖**那套脆弱的终端清洗管线
> （`cleanTtyOutput` / `extract*AssistantText` / `is*TuiLine` 等启发式），讨论组路径彻底绕开它。

```
acg serve                                     # 启动 web server（应用入口，给用户用）
acg say --next <成员名> "我的发言内容"      # 普通成员/主理人：提交发言并提名下一位
acg say --next <成员名> < message.txt        # 长内容走 stdin
acg end ["结论/收尾语"]                        # 仅主理人：提交收尾发言并结束讨论
acg whoami                                    # 调试：打印当前组/成员/角色
acg recap                                     # 可选：请求 server 回灌一份讨论摘要（早期上下文被压缩时的补救）
```

**提名自己 ✅【决策 7：默认拒绝】**：`--next` 不可指向自己，server 校验并拒绝，防自说自话刷轮次；
agent 想继续说就在同一次发言里说完。

对应 server 端 REST（新增，风格对齐现有 `server.mjs:107-151`）：
```
POST /api/sessions/:sessionId/members/:memberId/say   { content, next }
POST /api/sessions/:sessionId/members/:memberId/end   { content? }   // server 校验该 member 是否 host
GET  /api/sessions/:sessionId/recap
# 另需：创建/启动/停止/派生新轮/异常恢复/关闭话题讨论
POST /api/groups/:groupId/sessions   { topic, maxRounds }   // 发起一个话题讨论
POST /api/sessions/:sessionId/start                          // 拉起成员会话 + host 首发
POST /api/sessions/:sessionId/stop                           // 强制结束（人工兜底，会话仍保留存活）
POST /api/sessions/:sessionId/revisions { feedback, mode, cwd } // 用户反馈驱动的派生讨论
POST /api/sessions/:sessionId/reopen                         // 仅异常中断恢复，非正常结果页入口
POST /api/sessions/:sessionId/close                          // 显式关闭：kill 该 session 全部成员 PTY
```

> 这一调用同时是**"本轮发言结束"的可靠信号**——比现有靠启发式猜测 agent 是否说完的 `isAgentReadyForInput`
> 状态机可靠得多，是本设计的关键齿轮。

## 5. 上下文同步策略 ✅【决策 4：仅增量 + 只传"别人"的发言】

由于 TTY 启动的 agent **自带记忆**（自己的 CLI 会话保留自己说过的话），采用最省 token 的最优解：

- agent 自己说过的话 → **永不回传**（它自己上下文里已有）。
- 每次轮到某成员，server **只注入"自该成员上次发言以来、其他成员说的新消息"**（按 `seq` 计算增量）。
- 这些增量会**累积**进该成员自己的上下文 → 它实际看到了完整讨论过程，但每条别人的话**全程只传一次**。
- 等价于"完整记录"与"仅增量"两个选项优点的合并，代价几乎为零。

**兜底**：
1. 开场首轮注入必须包含 `话题 + 人设 + 职责 + 规则`。
2. server 保证不丢轮（见 3.2 丢轮防护）。
3. 某成员 CLI 自身压缩/截断了早期上下文时，主理人可触发 / 成员可 `acg recap` 让 server 回灌一份摘要。

注入给"下一位"的 prompt 结构（示意）：
```
【讨论继续】话题：<topic>
你的人设：<persona>　你的职责：<duty>
自你上次发言以来的新发言：
  <成员A>：……
  <成员B>：……
请发言，然后调用：acg say --next <你提名的下一位> "你的发言"
（若你是主理人且认为讨论可结束，改用：acg end "收尾"）
```

## 6. 前端 / UI ✅【决策 10：复用三栏 + 成员终端标签页】

布局延续现有三栏，最大化复用：
```
┌─树─┐┌─讨论会话流──┐┌─终端────┐
│▾电商 ││ 主理人：...   ││[A][B][C]   │
│ #话题││   张三：...  ││ $ ...      │
│  A  ││ 李四：...    ││ $ ...      │
│  B  ││   王五：...  ││ 当前发言人 A│
│  C  ││[输入/控制条]││            │
└────┘└───────────┘└────────┘
```
- **左侧项目树**：项目下新增「讨论组」分组节点，与 Agent 平级；组下挂多个「话题讨论」子节点，各自展示状态点（idle/running/ended）。
- **中栏 = 讨论会话流**：多 agent 讨论渲染成一条**按成员着色/带人设标识的会话流**（每条消息归属到对应成员），区别于现有单 agent 的 ChatPanel。底部控制条：开始 / 强制结束 / 关闭会话、当前发言人、轮次进度。正常结束后显示「我有不同意见」，不显示常驻的「重开/恢复」。
- **右栏 = 成员终端标签页**：按成员（A/B/C…）切换标签，查看该话题讨论下某成员会话的原始 PTY 输出；高亮"当前发言人"。
- **创建/编辑讨论组表单**（模板，不含话题）：组名、规则、成员列表（每个含 name/runtime/model/persona/duty + 主理人单选）。校验：主理人有且只有一个；成员名组内唯一。
- **发起话题弹窗**：选定讨论组 → 填写 topic、maxRounds → 创建并启动一个话题讨论。

> UI 具体视觉以 `style.png` 风格延续；落地前若视觉源不清晰，按 `agent-console/AGENTS.md` 走 Product Design 的 `get-context`。

## 7. 与现有架构的衔接清单

- **打包 / 分发（决策 6）**
  - 把项目打成可全局安装的 npm 包，`package.json` 增加 `bin: { acg: ... }`。
  - `acg serve` = 现在的 `node server.mjs`（应用入口）；`acg say/end/...` = 讨论 CLI 子命令。
  - CLI 通过 `AGENT_CONSOLE_API` 回调本地 server；身份靠 PTY 注入的 SESSION_ID/GROUP_ID/MEMBER_ID/TOKEN。
- `server.mjs`
  - 新增运行态（内存）：话题讨论 session 的当前发言人/轮次、各成员 PTY 会话句柄。
  - 新增**持久化（决策 9）**：讨论组模板 / 话题讨论 / DiscussionMessage 落 **JSON 文件**，启动时加载、变更时写回。
  - 新增 `/api/sessions/...`（say/end/recap/start/stop/reopen/close）+ `/api/groups/:id/sessions`（发起话题）REST。
  - 复用 `startPtySession`，但 session key 带 **sessionId + memberId** 维度；env 注入扩展 SESSION_ID/GROUP_ID/MEMBER_ID/API/TOKEN。
  - 讨论组路径**不经过** `cleanTtyOutput` 清洗链。
- `App.tsx`
  - 讨论组/话题/消息数据**从 server API 读取**（非 zustand persist）；现有 projects/agents/commands 仍走前端本地，互不影响。
  - 新增创建/编辑讨论组弹窗、发起话题弹窗、话题讨论主视图（含重开/关闭会话）、成员终端标签页。
- 运行时元数据复用现有 `runtimeMeta`（server）与 `agentRuntimeOptions`（App.tsx），新增 runtime/model 时两处仍需同步（既有约定）。

## 8. 已定决策汇总

| # | 决策点 | 结论 |
|---|--------|------|
| 1 | 发言内容如何进入系统 | **CLI 显式携带内容**（`acg say "..." --next X`），绕开终端清洗管线 |
| 2 | 发言调度与结束 | **agent 提名下一位 + 主理人随时 end + maxRounds 硬上限** |
| 3 | 成员实例模型 | **拉起专属 PTY 会话**，与项目独立 Agent 隔离 |
| 4 | 上下文同步 | **仅增量、只传别人的发言**，依赖 agent 自带记忆累积完整过程 |
| 5 | 讨论组与话题关系 | **解耦**：组是可复用模板；每个话题是独立 DiscussionSession，**各自新开成员会话**，话题间记忆隔离、可并发 |
| 6 | CLI 分发 | **全局 npm bin `acg`**，自带 `acg serve` 启动 web server（应用入口） |
| 7 | 提名自己 | **默认拒绝** `--next` 指向自己 |
| 8 | 会话回收 | **结束后保留存活**；提供**显式"关闭会话"**主动 kill；原 session 不再原地重开 |
| 9 | 持久化 | **server 落盘 JSON**（讨论组/话题/消息）；前端经 API 读取，偏离"全本地"现状 |
| 10 | UI 布局 | **复用三栏**：中栏讨论会话流、右栏成员终端标签页 |
| 11 | 结果不满后的继续讨论 | **用户反馈驱动派生 session**；单入口、线性轮次、父 session 不可变；异常恢复与继续讨论分离 |

## 9. 结果不满后的继续讨论（待实现）

### 9.1 用户故事与成功标准

**用户故事**：作为讨论发起人，当我对已结束的结论不满意时，我希望先说明自己的意见，再让全组围绕这条意见开启新一轮讨论，而不是只和某个 Agent 私聊。

**成功标准**：

- 用户不接触 session、PTY 或发言权等实现概念，即可启动新一轮；
- 用户意见进入权威消息记录，主理人收到正式开场提示并能通过 `acg say` 交给下一位；
- 新旧轮次状态、消息和结论独立，刷新或服务重启后仍能还原；
- 新结论能说明相对上一轮的保留项、修改项、废弃项及依据。

### 9.2 交互流程

1. 正常结束页只显示一个主入口「我有不同意见」；「恢复讨论」只在系统检测到异常中断时上下文化出现。
2. 点击后在结论下方就地展开并聚焦反馈框，不跳转、不弹出独立设置页。
3. 输入区文案为「你希望这一轮重点解决什么？」；反馈必填，MVP 使用固定示例，不依赖模型生成占位。
4. 默认为基于现有结论修订；仅保留次要选项「不沿用上轮结论，从头讨论」。不提供「修订／推翻／追问」三选一。
5. 提交后立即创建派生 session，时间线插入「我的意见」，切换到新一轮并展示主理人正在回应。
6. 新一轮结束后，按「第 1 轮结论 → 我的意见 → 第 2 轮结论」的线性时间线展示；旧逐字记录默认折叠。
7. 已结束讨论的成员终端默认只读，并引导用户通过「我有不同意见」提交。不将终端原始输入自动转译为用户反馈。

### 9.3 服务端原子流程

`POST /api/sessions/:sessionId/revisions { feedback, mode, cwd }` 需原子完成：

1. 校验父 session 已结束、反馈非空，且父 session 是当前线性链的最新一轮；
2. 创建带 `parentSessionId` 和 `revisionNo` 的派生 session；
3. 写入 `user_feedback` 消息，展示作者为「我」，不伪装成组员发言；
4. 保存上一轮结论快照，旧数据无结构化结论时，以最后一条主理人收尾消息作为兼容来源；
5. 初始化当前发言人为主理人，拉起成员 PTY，并向主理人注入新轮开场提示；
6. 任一关键步骤失败时不得留下「有 session 无反馈」或「有反馈未启动」的半成功状态；
7. 防止重复点击和并发派生：同一父链不能同时创建两个最新修订，接口需幂等或具备防重复约束。

PTY 是否在派生轮次间复用需先做技术验证。无论复用还是重建，都必须注入正式的新轮提示，不得依赖 PTY 自身记忆来表示新讨论已启动。

### 9.4 Agent 提示规则

主理人的新轮开场提示至少包含：

- 上一轮结构化结论；
- 用户原始反馈，不用模型改写替代；
- 本轮是基于结论修订，还是从头讨论；
- 要求先指出反馈挑战了哪些假设或决策，再列出本轮待解决问题并提名下一位；
- 明确用户观点是新证据或约束，可以被分析和质疑，不是预设答案。

收尾提示要求输出保留项、修改项、废弃项、修改依据、仍有分歧和待验证事项。如模型未稳定生成对照，UI 降级展示为普通「本轮结论」，不展示空栏目。

### 9.5 边界、验收与待验证

**MVP 边界**：

- 只允许从最新一轮继续，不做分支或版本树；
- 沿用原讨论组成员，不在主路径提供成员调整、轮次设置或定向追问模式；
- 已结束成员终端默认只读，高级调试解锁后置。

**验收项**：

- 正常结束页不再展示容易误解的「重开」按钮；
- 提交反馈后存在 `user_feedback` 记录，主理人收到正式新轮提示，并可正常交权；
- 父 session 保持 `ended`，派生 session 拥有独立状态、轮次和消息；
- 已结束终端只读并给出正确引导；
- 刷新或服务重启后仍能还原线性讨论轮次；
- 结果优先展示变化对照，生成失败时可降级为普通结论；
- 覆盖空反馈、重复提交、启动失败回滚、PTY 存活/重建、服务重启恢复和非最新轮派生等测试。

**待验证**：

- 入口文案「我有不同意见」与「继续讨论」的理解率；
- 用户能否在不接触终端的情况下完成新一轮启动；
- 复用存活 PTY 是否会被旧对话状态干扰，是否应统一重建；
- 主流模型生成「保留／修改／废弃」对照的稳定性；
- 历史会话无结构化结论时的快照兼容质量。

4. Agent 协议：新增新轮开场与对照式收尾提示模板，不改变普通 `acg say` 交接协议。
5. 测试：覆盖派生链、接口原子性、幂等、失败恢复和旧数据兼容。

# 自动开发编排功能需求文档（PRD）

> 产品：Agent Console  
> 状态：待评审 / 待实现  
> 版本：v1.2  
> 日期：2026-06-22  
> 配套阅读：`讨论组功能需求文档.md`、`CLAUDE.md`

## 1. 背景

Agent Console 已具备讨论组能力：一个讨论组包含多个带人设和职责的成员，每个成员拥有独立 PTY，会话由 CLI 显式提交发言并完成成员间交接。

当前讨论组适合开放式讨论，但不能可靠完成一项长周期开发目标。用户仍需人工拆分需求、切换 Agent、创建分支、传递 Review 意见、运行测试和完成集成。

本需求在讨论组基础上增加“流程运行”能力。用户提交一个开发目标后，系统按照可复用模板组织多个 Agent，完成规划、开发、审查、测试和集成。角色与流程不固定，测试、文档、安全审查等角色可以按模板启用或移除。

## 2. 产品目标

### 2.1 核心目标

1. 用户只需说明目标、选择仓库和最终集成方式，即可启动一次自动开发运行。
2. 复用讨论组的成员配置、独立 PTY、消息记录和终端观察能力；流程运行按任务创建执行会话，不要求一个开发 Agent 的 PTY 贯穿整个运行。
3. 通过角色模板、流程模板和阶段提示词模板支持不同开发方法，不在代码中固定角色数量与流转顺序。
4. 支持开发、Review、修改、测试、提交和最终集成的自动循环。
5. 所有关键步骤可观察、可暂停、可恢复、可人工接管、可审计。
6. 不覆盖用户已有改动，不因暂停、失败或终止丢失已产生的分支和提交。

### 2.2 非目标

首版不包含：

- 多个子任务并行开发；
- 跨多个仓库的原子变更；
- 从零搭建任意流程图；
- 仅依赖自然语言提示词控制流程；
- Review Agent 默认直接修改代码；
- 自动强推、绕过受保护分支或静默执行高风险 Git 操作；
- Agent 静默扩大需求范围；
- 自动处理账号认证、密钥或凭据。
- 依赖 `/compact` 等 runtime 私有命令维持长流程上下文。

## 3. 产品原则

1. **讨论组是名册，流程模板是打法。** 同一讨论组可以在不同运行中选择不同流程。
2. **角色可组合，不固定为四个。** 规划、开发、Review、测试、合并仅是默认角色集合。
3. **提示词负责指导，引擎负责约束。** Agent 理解任务依赖提示词，流程合法性由服务端验证。
4. **数据模型可分层，用户入口保持简单。** 普通用户选择一套“运行方式”，不需要理解模板内部结构。
5. **终端是证据，不是主界面。** 默认展示结构化进度，原始 PTY 按需展开。
6. **已启动运行使用不可变模板快照。** 全局模板变更不得影响运行中的任务。
7. **角色配置复用，执行会话按任务隔离。** 开发角色在同一任务的开发与修改循环中复用会话，任务通过后关闭会话，下一任务创建新会话。
8. **持久化产物是长期记忆。** PTY 上下文只服务当前任务；跨任务信息通过结构化交接产物传递，不依赖完整聊天记录或 runtime 上下文压缩。
9. **项目模板可覆盖，运行模板必须冻结。** 项目目录下的 `.acg` 可提供团队自定义角色、流程和提示词模板；创建运行时仍保存最终快照，后续文件变化不影响已启动运行。

## 4. 用户与核心场景

### 4.1 目标用户

- 希望将明确开发目标交给多个 Agent 协作完成的开发者；
- 需要保留审查、测试和人工确认门槛的团队成员；
- 希望定制团队开发流程和角色配置的高级用户。

### 4.2 核心场景

1. 用户选择“标准开发”模板，系统将目标拆成多个子任务，逐项完成开发、Review、测试与提交，最后创建 PR。
2. 用户选择“轻量开发”模板，不启用独立测试角色，由开发自测、Review 验证后提交。
3. 用户从标准模板微调，增加文档角色或调整最大 Review 轮次。
4. Review 或测试不通过，系统把结构化问题交还开发角色修改，直至通过或达到重试上限。
5. Agent、PTY 或服务中途退出，用户恢复运行后从最后一个已确认状态继续。
6. 用户在最终集成前查看成果、测试和未解决问题，决定创建 PR、确认直接合并或暂不集成。
7. 高级用户在仓库 `.acg` 目录中维护团队角色、流程和提示词模板，创建运行时直接选择项目模板或在项目模板基础上做本次调整。

## 5. 核心概念

### 5.1 讨论组（DiscussionGroup）

讨论组是可复用成员名册，继续保存成员的 runtime、model、persona、duty 等配置。讨论组可设置默认流程模板，但不与某个流程永久绑定。

### 5.2 角色模板（RoleTemplate）

角色模板定义一种岗位能力，而非一个正在运行的 Agent。

建议字段：

```ts
type RoleTemplate = {
  id: string;
  name: string;
  description: string;
  systemPrompt: string;
  allowedTools: string[];
  permissions: {
    canWriteCode: boolean;
    canRunCommands: boolean;
    canCommit: boolean;
    workspaceScope: "repository" | "worktree" | "read-only";
  };
  outputSchema: object;
  defaultRuntime?: string;
  defaultModel?: string;
  minInstances: number;
  maxInstances: number;
};
```

官方可提供规划、开发、代码审查、测试、集成、文档、安全审查等角色模板。

### 5.3 流程模板（WorkflowTemplate）

流程模板定义一次运行的阶段与流转规则。

```ts
type WorkflowTemplate = {
  id: string;
  name: string;
  version: number;
  description: string;
  requiredRoles: RoleBinding[];
  optionalRoles: RoleBinding[];
  stages: WorkflowStage[];
  transitions: WorkflowTransition[];
  defaultSettings: {
    maxReviewRounds: number;
    integrationMode: "pull_request" | "direct_merge";
  };
  createdAt: string;
  updatedAt: string;
};
```

每个阶段至少定义：

- 执行角色；
- 入口条件；
- 阶段提示词模板；
- 所需上下文；
- 允许的结构化结果；
- 成功、失败与退回路径；
- 超时和重试策略；
- 是否需要人工确认。

### 5.4 阶段提示词模板（StagePromptTemplate）

提示词按阶段动态生成，不使用一份贯穿全程的巨型提示词。

支持的首版变量至少包括：

- 用户目标；
- 仓库与目标分支；
- 当前任务及依赖；
- 范围、禁止改动和验收标准；
- 当前工作分支/worktree；
- diff 与变更摘要；
- 测试命令和结果；
- Review 发现与当前轮次；
- 最终集成方式。

每次交接只注入当前阶段所需信息。普通讨论继续使用“自上次发言以来的增量上下文”；流程运行根据当前任务、依赖任务交接产物和持久化阶段产物重新构建上下文，不回灌完整历史消息。

### 5.5 运行实例（WorkflowRun）

运行实例代表一次实际自动开发过程。

```ts
type WorkflowRun = {
  id: string;
  groupId: string;
  workflowTemplateId: string;
  workflowSnapshot: object;
  goal: string;
  repositoryPath: string;
  baseBranch: string;
  integrationMode: "pull_request" | "direct_merge";
  status: WorkflowRunStatus;
  currentStageId: string;
  currentTaskId?: string;
  settings: object;
  createdAt: string;
  updatedAt: string;
};
```

运行创建时保存角色和流程模板快照。运行期间修改全局模板，不影响该运行。

### 5.6 子任务（DevelopmentTask）

规划阶段必须产出结构化任务，而非仅返回自然语言列表。

```ts
type DevelopmentTask = {
  id: string;
  title: string;
  objective: string;
  scope: string[];
  forbiddenChanges: string[];
  dependencies: string[];
  acceptanceCriteria: string[];
  suggestedTests: string[];
  expectedFiles: string[];
  status: TaskStatus;
};
```

首版串行执行。调度器只能选择依赖已完成的下一项。

### 5.7 角色执行会话（RoleExecutionSession）

角色执行会话是某个成员在一次具体阶段或任务中的 PTY 实例，不等同于长期存在的角色配置。

```ts
type RoleExecutionSession = {
  id: string;
  runId: string;
  memberId: string;
  roleId: string;
  taskId?: string;
  stageId: string;
  status: "starting" | "running" | "completed" | "failed" | "terminated";
  startedAt: string;
  endedAt?: string;
  transcriptRef?: string;
};
```

首版默认会话策略：

- 规划角色的会话服务于本次规划，计划产物确认后即可关闭；
- 开发角色每个任务创建一个新会话；同一任务被 Review 或测试打回时，继续复用该任务的开发会话；
- 当前开发会话异常退出时，可基于任务、worktree、diff、测试结果和未关闭问题创建替代会话；
- 当前任务正式通过并提交后，关闭其开发会话；下一任务即使仍绑定同一成员，也必须创建新会话；
- Review、测试和集成等角色的会话范围由流程模板配置，首版至少支持按任务或按阶段创建。

流程运行不以 `/compact` 作为正常生命周期步骤。runtime 是否支持上下文压缩，不影响流程正确性和恢复能力。

### 5.8 任务交接产物（TaskHandoffArtifact）

任务通过后，系统必须在关闭开发会话前生成结构化交接产物，作为后续任务和异常恢复的上下文来源。

```ts
type TaskHandoffArtifact = {
  id: string;
  runId: string;
  taskId: string;
  commitSha: string;
  integrationBaseline: string;
  changeSummary: string;
  changedFiles: string[];
  decisions: string[];
  changedInterfaces: string[];
  testResults: object[];
  closedFindings: string[];
  unresolvedIssues: string[];
  createdAt: string;
};
```

交接产物由服务端根据已验证的 commit、diff、测试和 Review 结果生成确定性字段，Agent 只补充设计决策、接口变化和已知限制。系统可以将其渲染为可读交接文档，但默认保存于运行记录，不要求向业务仓库写入额外 Markdown 文件。

## 6. 官方预设模板

### 6.1 轻量开发

流程：规划 → 开发 → Review → 任务提交 → 集成。

适合范围清晰、风险较低、仓库已有可靠自动测试的任务。开发角色负责自测，Review 角色检查代码和测试证据。

### 6.2 标准开发

流程：规划 → 开发 → Review → 测试 → 任务提交 → 集成。

Review 不通过或测试失败均退回开发。测试角色独立验证验收标准与回归风险。

### 6.3 后续模板

可增加高保障开发模板，在标准流程中加入架构审查、安全审查、文档或多级测试。该模板不作为首版发布阻断项。

## 7. 端到端流程

### 7.1 创建运行

启动前只要求用户填写：

1. 开发目标；
2. 仓库路径与目标分支；
3. 最终方式：创建 PR 或直接合并。

用户同时选择一套流程模板。讨论组默认模板可自动选中，但允许本次改选。

高级设置默认折叠，包括：

- 最大 Review/修改轮次，默认 3；
- 是否启用测试角色；
- 是否启用文档角色；
- 角色实例数；
- 自动探测到的测试命令；
- 超时与暂停策略。

创建前系统必须检查：

- 路径是有效 Git 仓库；
- 目标分支存在或可明确创建；
- 工作区是否存在用户未提交改动；
- 所需角色能否映射到可用成员/runtime；
- Git 与流程所需命令是否可用。

系统不得自动接管用户未提交改动。无法安全隔离时阻止启动并说明处理方式。

### 7.2 计划生成与确认

规划角色读取目标和仓库上下文，生成结构化子任务、依赖和验收标准。

计划确认页展示：

- 任务卡片及一句话目标；
- 预计涉及文件；
- 任务依赖和顺序；
- 每项验收标准；
- 自动探测到的测试命令；
- 预计启用的角色与流程摘要。

用户可以修改标题、调整无依赖冲突的顺序、删除、跳过任务或退回重新规划。

默认需要用户确认后开始。“全自动模式”可以跳过后续非高风险确认，但必须单独开启并展示风险；范围扩大和直接合并仍不得静默执行。

### 7.3 任务开发

每个任务创建独立分支：

```text
task/<run-id>/<task-id>
```

优先使用独立 worktree 隔离任务。开发角色只处理当前任务，并输出：

- 变更摘要；
- 修改文件；
- 自测命令和结果；
- 已知限制；
- 候选 diff/提交引用。

开发自检通过仅代表“可送审”，不代表任务完成。

调度器在任务开始时为开发角色创建新的执行会话，并注入：

- 总目标和当前任务对象；
- 当前集成基线、任务分支和 worktree；
- 当前任务直接或间接依赖中与本任务有关的交接产物；
- 已确认的跨任务设计决策和接口变化；
- 验收标准、允许范围、禁止改动和建议测试。

不得向新会话回灌全部历史发言、全部终端记录或所有已完成任务的完整 diff。上下文应按当前任务依赖和相关性裁剪。

### 7.4 Review 与修改循环

Review 角色只审查当前任务范围、diff、验收标准和测试证据，默认不直接改代码。

Review 输出必须结构化：

```ts
type ReviewResult = {
  verdict: "approved" | "changes_requested" | "failed";
  findings: Array<{
    id: string;
    severity: "blocking" | "major" | "minor";
    file?: string;
    line?: number;
    title: string;
    detail: string;
    suggestedFix?: string;
  }>;
};
```

- `approved`：进入模板定义的下一阶段；
- `changes_requested`：携带问题退回原开发角色和原任务分支；
- `failed`：进入失败处理或人工接管。

每轮展示 Review 问题及开发对应修改。同一问题连续两轮未解决时高亮。

达到上限前一轮，系统提前提示用户选择：

- 增加一轮；
- 转为人工处理；
- 跳过当前任务；
- 终止整个运行。

Review 或测试打回期间属于同一个任务生命周期，默认继续使用该任务现有的开发会话，保留其局部上下文。若该会话已经退出，则创建替代会话，并注入当前任务对象、worktree 状态、候选 diff、测试结果和全部未关闭发现。

### 7.5 测试阶段

测试阶段是否存在由流程模板决定。独立测试角色应获得验收标准、候选 diff、环境信息和允许的测试命令，但默认不改业务代码。

测试结果必须包含：

- 执行命令；
- 退出码；
- 通过/失败数量；
- 失败摘要；
- 日志或报告位置。

测试通过后进入提交；失败则按模板退回开发。禁止通过提示词或 Agent 自行忽略失败测试。

### 7.6 任务提交与集成基线

正式任务提交只能在模板要求的 Review、测试等门槛全部通过后生成，并记录 commit SHA。

正式提交生成后，系统必须：

1. 生成并持久化任务交接产物；
2. 更新后续任务使用的集成基线；
3. 标记当前任务的角色执行会话已完成并关闭其 PTY；
4. 为下一任务创建全新的开发执行会话；
5. 从持久化状态构建下一任务提示词，而不是延续上一任务的 PTY 上下文。

后续任务基于已通过任务形成的集成基线继续开发，避免所有任务完成后才集中处理冲突。

如目标分支在运行期间发生变化，系统必须暂停并重新验证。rebase/merge 和冲突处理过程必须对用户可见，不得在旧基线上继续静默集成。

### 7.7 最终集成

全部子任务完成后，由模板指定的集成角色执行：

- 全量测试；
- 未关闭 Review/测试问题检查；
- 提交链完整性检查；
- 目标分支漂移检查；
- 最终变更摘要生成。

最终确认页展示：

- 变更文件和总 diff；
- 各任务及 commit；
- 全量测试结果；
- 未关闭问题；
- 目标分支与集成方式。

默认操作为创建 PR。直接合并必须在最终动作前二次确认。受保护分支不得直接合并，操作置灰并说明原因。

存在以下任一条件时禁止集成：

- 全量测试失败；
- 阻断级 Review 问题未关闭；
- 提交链缺失或无法验证；
- 目标分支漂移未处理；
- 用户未完成要求的高风险确认。

## 8. 流程引擎

### 8.1 运行状态

建议基础状态：

```text
draft
planning
awaiting_plan_approval
running
paused
needs_attention
integrating
completed
failed
terminated
```

任务阶段由流程模板定义，例如：

```text
pending → developing → reviewing → testing → approved → committed
                         ↘ fixing ↗     ↘ fixing ↗
```

### 8.2 结构化交接

普通讨论继续使用：

```text
acg say --next <成员名> "内容"
```

流程运行需要新增结构化阶段结果命令或等价 API。具体命令名由技术设计确定，能力至少覆盖：

- 提交阶段结果；
- 附带结构化状态和产物；
- 查询当前运行、任务和阶段；
- 请求暂停或人工处理；
- 获取当前阶段提示词/上下文回顾。

在流程模式中，下一执行角色由流程引擎依据模板决定，而不是完全由 Agent 自由提名。

阶段结果提交成功不代表复用当前 PTY进入下一任务。流程引擎同时负责：

- 判断当前任务是否仍处于开发—Review—修改循环；
- 当前任务未结束时将问题投递回该任务的开发会话；
- 当前任务通过后生成交接产物并终止任务会话；
- 创建下一任务的新执行会话并注入裁剪后的上下文。

### 8.3 非法迁移

服务端必须拒绝非法迁移，例如：

- 开发完成后跳过 Review；
- Review 未通过却进入测试或提交；
- 测试失败却标记任务通过；
- 非集成角色尝试最终合并；
- 超过重试上限后继续自动循环。

拒绝后向 Agent 和 UI 返回可读原因，例如：“当前流程要求开发完成后先经过 Review”。不得静默纠正。

## 9. 模板配置体验

### 9.1 普通用户

启动页只展示“运行方式”卡片，每张卡包含：

- 名称和一句话说明；
- 角色头像/数量；
- 流程摘要；
- 最大重试轮次；
- 是否包含独立测试；
- 默认集成方式。

普通用户不需要理解角色模板、流程模板和提示词模板之间的关系。

### 9.2 参数微调

首版允许通过表单和开关微调：

- 是否启用测试角色；
- 是否启用文档角色；
- 最大重试轮次；
- 角色实例数；
- PR 或直接合并。

系统应自动调整受影响阶段，并在启动前展示最终流程摘要。

### 9.3 高级编辑

首版不提供可视化流程图编辑器。阶段增删、迁移条件和提示词编辑可在后续版本通过“阶段列表 + 规则表单”开放。

模板编辑需校验：

- 每个阶段有可用执行角色；
- 至少存在一个合法完成路径；
- 不存在无退出条件的自动循环；
- 可选角色被移除后，不残留不可达阶段；
- 高风险动作存在人工确认或明确策略。

### 9.4 项目级本地模板覆盖

系统支持在目标仓库根目录创建 `.acg` 文件夹，用于保存项目级开发编排模板。项目模板面向高级用户和团队协作，可进入业务仓库版本控制，作为系统内置模板的替换或补充来源。

建议目录结构：

```text
.acg/
  roles/
    planner.json
    developer.json
    reviewer.json
    tester.json
    integrator.json
  workflows/
    lightweight-dev.json
    standard-dev.json
  prompts/
    planning.md
    development.md
    review.md
    testing.md
    integration.md
```

模板来源优先级：

```text
系统内置模板
  < 用户全局模板（后续可选）
  < 项目 .acg 模板
  < 创建运行时的本次修改
  < 暂停后的运行级模板修订
```

首版至少支持：

- 读取 `.acg/roles/*.json` 作为角色模板；
- 读取 `.acg/workflows/*.json` 作为流程模板；
- 读取 `.acg/prompts/*.md` 作为阶段提示词模板；
- 通过模板 ID 覆盖同名系统内置模板；
- 在创建运行页标记模板来源：`内置`、`项目 .acg`、`本次修改`；
- 启动前展示最终流程摘要，并提示存在项目级覆盖；
- 创建运行时将最终合成后的角色、流程和提示词保存为不可变运行快照；
- 在快照中记录模板来源、相对路径、版本和内容 hash，便于审计和恢复。

创建运行时修改属于本次运行覆盖。用户可以在启动页调整参数或提示词，但默认只影响当前 `WorkflowRun.workflowSnapshot`，不自动写回 `.acg`。如需复用，必须由用户显式执行“保存为项目模板”或后续模板管理功能。

项目模板必须通过与内置模板相同的 schema 和安全校验。以下行为必须被拒绝：

- 通过提示词或模板配置跳过服务端强制的 Review、测试、提交和集成门槛；
- 将受保护分支直接合并、强推、清理 worktree 等高风险动作设为静默自动执行；
- 引用仓库 `.acg` 目录外的任意提示词文件，或通过路径穿越读取仓库外文件；
- 将角色权限提升到超出服务端策略允许的范围；
- 在模板中包含明显的密钥、token 或凭据内容。

项目 `.acg` 文件只在创建运行、刷新模板列表、保存模板和运行级修订确认时读取。已启动运行不得直接引用磁盘上的模板文件继续执行，必须使用快照内容。

## 10. 运行总览与控制

### 10.1 总览信息

主界面常驻展示：

- 当前任务序号/总数；
- 当前阶段与 Review 轮次；
- 整体进度；
- 每个任务的状态；
- 当前活动 Agent；
- 已耗时；
- 最近输出时间；
- 使用的模板快照版本；
- 模板来源及项目 `.acg` 覆盖状态；
- 当前角色执行会话及其所属任务。

长时间没有新输出时显示“疑似卡住”，提供查看终端、重试和暂停入口。

### 10.2 暂停、恢复与终止

- 暂停按钮全程可见；
- 默认采用安全暂停：当前 Agent 完成一个完整阶段结果后停止；
- 必要时提供强制停止，但需提示可能丢失当前未落盘输出；
- 恢复时说明将从哪个任务、阶段和模板快照继续；
- 恢复不得要求原 PTY仍然存活；系统可创建新执行会话，并从任务状态、Git 产物和结构化交接产物恢复；
- 终止默认保留分支、提交和运行记录；
- worktree 是否清理由用户选择，不自动删除未集成成果。

### 10.3 运行级模板修订

全局模板修改不影响当前运行。用户要修改当前流程时必须：

1. 暂停运行；
2. 创建运行级修订；
3. 查看角色、阶段和后续路径变化；
4. 确认从指定状态恢复。

界面明确显示：“当前运行使用模板 X 的快照 vN”。

## 11. 异常与人工接管

失败信息必须包含：

- 失败任务；
- 失败阶段；
- 原因摘要；
- 当前分支/worktree；
- 已有提交；
- 未关闭问题；
- 建议操作。

可用操作：

- 重试当前阶段；
- 退回开发；
- 增加一次重试；
- 转为人工处理；
- 跳过任务并继续；
- 终止整个运行。

跳过有依赖任务时，系统必须提示受影响的后续任务，并禁止形成不满足依赖的执行顺序。

## 12. Git 与安全要求

1. 不覆盖、暂存或提交运行开始前已有的用户改动。
2. 每个任务使用独立分支，优先使用独立 worktree。
3. 所有正式提交记录关联的任务、验收结果和 Review/测试证据。
4. 禁止 Agent 强推受保护分支、修改凭据或操作仓库外目录。
5. 破坏性 Git 命令必须由服务端策略阻止，不能只依赖提示词约束。
6. 直接合并默认关闭；启用后仍需最终确认。
7. 日志、提示词和结构化产物不得保存 token、密钥等敏感信息。

## 13. 持久化与审计

服务端必须持久化：

- 运行配置和模板快照；
- 子任务与状态迁移；
- 阶段执行角色；
- Agent 输入输出摘要；
- 分支、worktree 和 commit SHA；
- 执行命令、退出码和测试结果；
- Review 发现、轮次和关闭情况；
- 人工确认、暂停、恢复和终止操作；
- 失败原因与恢复点；
- 每个角色执行会话的任务/阶段归属、起止状态和终端记录引用；
- 每个已通过任务的结构化交接产物；
- 模板来源、相对路径、版本、内容 hash 和最终运行快照。

PTY 自身记忆不是恢复依据。服务或 Agent 重启后，系统必须依据持久化状态重建当前阶段上下文。

## 14. 功能优先级

### P0：首版必须

- 流程运行模式与模板快照；
- 轻量、标准两套官方模板；
- 项目 `.acg` 角色、流程和提示词模板读取、覆盖、校验与快照固化；
- 角色动态绑定，不固定角色数量；
- 结构化子任务和计划确认；
- 串行调度；
- 独立任务分支/worktree；
- 开发—Review—修改循环；
- 可选独立测试阶段；
- 流程引擎迁移校验；
- PR/直接合并策略与最终确认；
- 运行总览、暂停、恢复、终止；
- 状态、产物和审计记录持久化；
- 开发角色按任务创建执行会话，同一任务修改循环复用、任务通过后关闭；
- 任务交接产物生成及下一任务最小上下文注入。

### P1：首版后增强

- 文档、安全、架构审查等更多官方角色；
- 运行级模板修订；
- 更完整的模板参数微调；
- 模板差异对比、提示词编辑和“保存为项目模板”；
- 阶段列表和迁移规则表单编辑；
- PR 描述、变更日志和发布说明模板。

### P2：远期

- 子任务并行调度；
- 多开发 Agent 冲突协调；
- 可视化工作流编辑器；
- 模板市场与团队共享；
- 多仓库运行。

## 15. MVP 验收标准

### 15.1 预设模板

1. 轻量模板能够完成规划、开发、Review、任务提交和最终集成。
2. 标准模板能够经过独立测试角色；测试失败时正确退回开发。
3. 标准模板关闭测试角色后，流程自动移除测试阶段且不存在无效迁移。
4. 增加文档角色后，文档阶段能够按配置进入并完成。

### 15.2 端到端任务

使用一个干净测试仓库和包含 3 个依赖子任务的目标：

1. 系统生成可编辑的结构化计划；
2. 三个任务按依赖串行执行；
3. 至少注入一次 Review 打回和一次测试失败；
4. 开发角色收到结构化问题并完成修复；
5. 只有 Review/测试通过后才生成正式任务提交；
6. 最终可创建 PR，或在二次确认后直接合并非受保护分支；
7. 每个任务使用独立的开发执行会话；Review/测试打回时仍回到当前任务会话，任务通过后下一任务创建新会话。

另使用一个包含 20 个串行依赖子任务的测试运行验证：

1. 任一开发执行会话只包含一个任务及其修改轮次；
2. 后续任务可从结构化交接产物获得必要的接口和设计决策；
3. 不调用 `/compact`，也不回灌完整运行历史；
4. 第 20 个任务仍能基于正确集成基线完成开发、Review 和提交。

### 15.3 流程约束

以下非法操作必须被拒绝并显示可读原因：

- 跳过 Review；
- 测试失败后直接提交；
- 非集成角色执行合并；
- 超过最大重试轮次继续自动循环。

### 15.4 恢复与安全

1. 分别终止开发 Agent、Review Agent、测试 Agent 和服务进程，运行均可从最后一个已确认状态恢复。
2. 运行中修改全局模板，当前运行仍使用原快照。
3. 终止运行后，已有分支和提交仍存在。
4. 仓库含用户未提交改动时，系统不会覆盖、暂存或提交这些改动。
5. 目标分支漂移、全量测试失败或存在阻断问题时，最终集成被阻止。

### 15.5 项目模板覆盖

1. 测试仓库包含 `.acg/workflows/standard-dev.json` 时，创建运行页优先展示项目版本并标记来源。
2. 项目模板通过同 ID 覆盖内置标准模板后，运行快照使用项目模板内容。
3. 启动运行后修改 `.acg` 文件，当前运行仍使用启动时保存的快照。
4. 创建运行时临时修改最大 Review 轮次或阶段提示词，只影响本次运行快照，不自动写回 `.acg`。
5. 包含非法迁移、越权权限、路径穿越或高风险静默 Git 动作的 `.acg` 模板被拒绝，并显示可读原因。
6. 运行记录中可查看模板来源、相对路径、版本和内容 hash。

## 16. 成功指标

首版上线后关注：

- 运行启动成功率；
- 计划确认后无需人工复制上下文的任务占比；
- Review/测试回退后成功修复率；
- 异常中断后的恢复成功率；
- 自动运行最终形成有效 PR/合并的比例；
- 因非法迁移、Git 风险或未解决问题而被正确阻止的次数；
- 用户人工接管发生的阶段和原因。

## 17. 风险与对策

| 风险 | 对策 |
|---|---|
| 提示词漂移导致 Agent 跳过流程 | 服务端状态机验证结构化结果和迁移 |
| 模板概念过多 | 普通入口只展示“运行方式”，内部结构按需展开 |
| Review/开发反复打转 | 默认轮次上限、提前预警、重复问题高亮、人工接管 |
| 用户改动被覆盖 | 启动前检查、独立 worktree、禁止接管已有改动 |
| 目标分支运行期间变化 | 集成前漂移检测并暂停处理 |
| PTY 退出导致上下文丢失 | 状态和产物服务端持久化，按阶段重建上下文 |
| 长任务链导致开发 Agent 上下文膨胀 | 每个任务创建新开发会话，通过结构化交接产物传递必要信息，不依赖 `/compact` 或完整历史回灌 |
| 交接文档遗漏或描述失真 | commit、diff、测试和 Review 字段由服务端生成，Agent 仅补充设计决策与已知限制 |
| 模板更新影响正在运行的任务 | 每次运行保存不可变模板快照 |
| 项目 `.acg` 模板绕过流程或提升权限 | 模板 schema 校验、服务端策略兜底、非法迁移拒绝和高风险动作强制确认 |
| 项目 `.acg` 提示词引用仓库外文件或泄露敏感信息 | 限制读取范围、禁止路径穿越、启动前扫描明显凭据并提示用户处理 |
| 配置能力过度复杂 | 首版仅支持预设选择和安全参数微调 |

## 18. 待技术设计确认

以下内容不阻塞 PRD 评审，但需在技术设计阶段确定：

- 流程模板和运行快照的具体存储结构；
- 结构化阶段结果对应的 CLI 命令及 REST API；
- 临时候选变更使用未提交 diff、临时 commit 还是独立引用；
- worktree 创建、清理和异常恢复策略；
- GitHub/GitLab/其他代码托管平台的 PR 接口范围；
- 测试命令自动探测规则和执行沙箱；
- Agent 无输出的“疑似卡住”阈值；
- 模板版本升级和兼容策略；
- `.acg` 目录 schema、文件命名、模板 ID 冲突、覆盖优先级和内容 hash 规则；
- 项目模板差异展示、提示词编辑和保存回 `.acg` 的交互边界；
- 普通讨论会话与流程运行在现有 UI/API 中的兼容方式；
- 角色执行会话的 PTY key、创建、关闭和 transcript 归档方式；
- 任务交接产物的 schema、相关性裁剪规则和可读文档渲染方式。
